# Backlog-Harness Gap Audit

**The vision (MANIFESTO.md):** throw an entire backlog at Orion — the issues ARE the spec — and get the
whole thing built to spec, with incompleteness and contradictions escalated to a human before code,
zero drift, and everything proven to the manifesto/design standard.

**Method:** six parallel auditors (one per vision dimension) read the current code; every claimed gap was
then independently, adversarially verified (verifiers instructed to refute) and deduped against the open
beads backlog. Severities below reflect verifier adjustments. Every gap carries file:line evidence.

Severity legend: **blocker** = the vision literally cannot happen without it · **major** = fails/degrades at
real-backlog scale or violates a manifesto commitment · **minor** = polish.


## 1. Backlog as Input

**State today:** Backlog-as-input is absent entirely: every entry surface (TUI conversation, `orion submit --non-interactive` stdin, `orion run`, `orion change "<intent>"`) accepts exactly one typed intent string, and internal/tracker is explicitly a ONE-WAY store-to-tracker JSONL projection that "never reads back" — there is zero code that reads a beads DB, issues.jsonl, or GitHub issues as input. The Context Store enforces a single in-flight work item (latest project + latest spec), the decomposer is a fixed 6-task template rather than a backlog consumer, and the task/spec model (title/status/file_scope/proof_id tasks; HTTP-shaped ResponseContract with Route/Port/TimeZone) cannot represent 50 dependency-ordered prose issues. or-97z already names the strategic fix (pull an epic's task DAG FROM beads); or-tcs.2, or-zn8, or-809 and or-3p5 cover adjacent pieces, but no issue covers intent queueing or backlog snapshot anchoring.

But every mode is thin in a way that produces manifesto-grade false passes: the obligation vocabulary can only express single-file, single-service Go HTTP request/response cases; mutation scoring is 4 hardcoded string mutants that silently no-op when inapplicable; the hazard tier is vacuously green for any project that never ratified an STPA model (which the flow never drives); and operability (3 a.m. test) is asserted by a template runbook, never validated against the artifact. The generating agent sees the full case set and the corpus is a deterministic function of it, so there is no held-out surface — the manifesto's "validation withheld from the agent" holds only for corpus bytes, not information. For a real backlog (the vision's input), most requirements cannot even become obligations, which is the single biggest blocker.

## 5. Surviving a Real Backlog (Scale, Robustness, Recovery)

**State today:** The or-tcs epic landed a genuinely parallel, proof-gated single-epic pipeline: bounded-concurrency cluster builds (ORION_MAX_AGENTS, default 3) in isolated worktrees, a serialized integration queue with post-merge re-proof and rollback, idempotent worktree re-creation (or-d3w), a durable SQLite WAL context store with a Recall/FactBundle resume primitive, and a resilient llmclient (timeout/retry/jitter/breaker) on the native Anthropic path.
